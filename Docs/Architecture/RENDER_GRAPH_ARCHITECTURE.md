# Render Graph Architecture

**Status:** Required design contract
**Date:** 2026-07-15

## Why This Is A Separate Contract

The render graph is shared infrastructure for most renderer phases, not one renderer pass. Keeping its construction, execution, synchronization, and verification contract here prevents the Phase 3 checklist from implying that lifetime data alone is an executable graph.

## Current Foundation

The checked Phase 3 construction item is a compiler-only logical graph. A pass declares read, write, or read-write access with queue, required state, and shader-stage intent; compilation validates those declarations, derives RAW/WAR/WAW edges and declared ordering constraints, rejects invalid handles, ambiguous duplicate uses, transient read-before-write, and cycles, then produces stable topological order. Resource first/last-use intervals, state barriers, and cross-queue transition records are calculated from that order, not registration order.

The compiler deliberately does not by itself:

- bind declared resources to imported or graph-owned RHI resources;
- record and execute pass callbacks;
- turn abstract transitions into RHI barriers or queue signal/wait commands;
- retire per-frame graph state and transient resources against GPU completion;
- drive the real scene viewport without a renderer consumer.

Those belong to the ordered Phase 3 execution-core, cross-queue, and real-workflow integration items. The construction compiler is an authority for intended logical dependencies and state; it is not an execution path on its own. The current renderer consumer builds a three-pass imported-resource Scene viewport graph on both claimed NVRHI backends: `Scene Viewport Graph Clear` writes renderer-owned color/depth outputs, `Scene Viewport Graph Raster` consumes the immutable prepared Scene frame and records its indexed draws, and `Scene Viewport Graph Output Handoff` transitions the color output to `ShaderResource` for the existing presentation bridge. The executor wraps every callback with its stable pass debug name, so those labels are visible to the backend capture/marker systems without giving Scene code a native API path. The per-frame graph currently owns stack-lifetime recording contexts and therefore waits for its final graphics completion before destruction; this is a safe bridge, not transient pooling or frame-overlap policy.

## Ordered CPU Preparation And Parallel Recording Prerequisites

The original Phase 3C multithreaded-render line was split into independently reviewable prerequisites P1 and P2, now integrated by the production Scene viewport path. Phase 3C prerequisite P1 is a real current consumer, not a generic scheduler layer: after the caller-affine Editor update publishes an immutable `SceneRenderSnapshot`, `Frame.PrepareSceneRaster` runs `Renderer::PrepareCurrentSceneRasterFrame` on the CPU task graph's worker lane. It reads only that immutable snapshot and atomically publishes one immutable `SceneRasterFrame`; the caller-affine viewport graph then consumes only a prepared frame with the exact snapshot frame index. A missing or stale publication fails the D3D12/Vulkan viewport path instead of silently falling back to direct preparation. Deterministic-single-thread executes that same named task and publication flow inline, making it an oracle for the worker mode. Only `NVRHIRenderBackend::RunVulkanSceneViewportRasterSmoke` prepares synchronously after publishing its synthetic snapshot, because that named standalone smoke deliberately runs outside an Application frame task; `RenderCurrentSnapshot` has no such fallback.

Phase 3C first requires an explicit expected-before RHI transition boundary for worker-pre-recorded texture and buffer barriers. A graph callback records only compiler-supplied `Before`/`After` states into command-list-local staging. At submission D3D12 and Vulkan compare every first expected state against the exact live wrapper/tracker state, reject a mismatch before native acceptance, and publish staged wrapper state only after native queue/NVRHI acceptance. Vulkan seeds NVRHI tracking locally on its command list and never mutates global wrapper state during recording. This is the narrow prerequisite that lets a dependent same-effective pass record before its producer is submitted without treating a stale barrier source as valid; cross-queue ownership acquire/release remains caller-recorded because it depends on the producer token.

Phase 3C prerequisite P2 establishes a narrow, pass-level declaration that a callback and all captured data are worker-recording-safe. The default is caller-affine: existing callbacks may capture mutable frame diagnostics, constant-buffer allocation, native presentation state, or backend objects and therefore receive no inferred worker eligibility. Eligible same-effective passes without ownership operations receive separate RHI contexts and pre-record together, including a dependent consumer; compiled order remains the only submission/publishing authority and supplies the producer token. An accepted prefix remains authoritative after a later failure, and no CPU wait is inserted between GPU queue submissions. The deterministic mode invokes the identical declared pass preparation/recording functions inline rather than a second implementation. RHI exact-device ownership, cross-queue release/acquire dependencies, exact-token transient/context retirement, stable debug labels, and callback exception behavior remain unchanged.

Phase 3C is locally qualified on the RTX 3080 Ti for D3D12 and Vulkan: both parallel and deterministic-inline runs emitted the corresponding `SceneRasterPreparationV1` lane marker, `RenderGraphRecordingV1` (worker `overlap=yes`, inline `overlap=no`, two worker-eligible passes, three compiled-order submissions), and production `SceneViewportRenderGraphV1` exact-byte equivalence to the direct reference. The same bounded runs retain the existing `RHIQueueDependencySmokeV1` GPU-wait/`cpuWaitBetween=no` evidence; the execution loop introduces no CPU completion wait between accepted submissions. Deterministic tests additionally prove stale expected-before rejection with no publication, default affinity, bounded independent overlap, same-effective dependency-token ordering, cross-queue caller deferral, failure-prefix disposal, and exact retirement/reuse. This completes the stated local D3D12/Vulkan multithreaded render-preparation and command-recording line, not another backend/device qualification.

## Construction And Scheduling Contract

Each frame builds a graph from pass registrations. Every pass declares its resource reads and writes, required states, and queue class before compilation. Compilation must:

1. Validate resource handles, states, imported-resource contracts, and read/write hazards.
2. Derive dependency edges from declared data hazards rather than relying on registration order.
3. Reject cycles and unresolved read-before-write dependencies with actionable diagnostics.
4. Produce a deterministic topological pass order.
5. Compute first and last use in that resolved order.
6. Derive same-queue barriers and cross-queue ownership/synchronization transitions.

Execution must bind logical resources to physical `Engine::RHI` resources, invoke pass callbacks with a restricted execution context, record the compiled barriers and commands, and submit work in dependency order. Imported resources must declare initial and required final states. Queue transitions must include the required signal/wait relationship; a queue-name change alone is not synchronization.

Construction, single-queue execution core, cross-queue execution, and real-workflow adoption are separate roadmap gates. A compiled dependency/barrier plan does not complete execution. The core first binds real physical/imported resources, records same-queue work, and retires frame contexts by GPU completion; the next gate turns queue-dependency records into explicit signal/wait submission; the final gate moves a representative multi-pass Scene viewport onto that path and proves output equivalence before its bootstrap path is removed.

GPU-retired frame-context reuse depends on a backend-neutral completion boundary that did not exist when construction was completed: the original device contract exposed only synchronous `SubmitAndWait`, with no opaque submission token, completion query, or reusable recording lifecycle. Phase 3 therefore requires an explicit RHI prerequisite before the executor. `Device::Submit` returns after queue acceptance with a device-owned monotonic `CompletionToken`; `QueryCompletion` is nonblocking and `WaitForCompletion` has an explicit finite deadline. Tokens contain only opaque RHI device/submission identities, while each backend retains the corresponding native primitive. Tokens with zero identities, another device identity, or an unissued submission identity are invalid; completed tokens remain queryable for the current device lifetime.

Historical token validity does not require retaining one native completion record per submission. The D3D12 adapter keeps native queue/fence/timestamp state only for the live nonterminal suffix, compacts contiguous terminal identities into a scalar prefix, and retains explicit failed identities so their status cannot change. Compaction occurs before issuing later work, never by polling the just-published token: query-pool clients must still observe their accepted generation as Pending until an explicit query or wait establishes retirement. Retired tokens remain valid dependencies and skip redundant native waits; an incomplete dependency must still resolve to its retained queue/fence record. This bounds ordinary successful-session state without weakening zero, foreign, unissued, delayed-query, ownership, or cross-queue rules. Failed identities may remain sparse for the device lifetime because forgetting them would incorrectly turn a prior failure into success.

D3D12 assigns each submitted list a per-queue fence value and retains that mapping behind the token. Vulkan submits through the existing NVRHI device and maps each returned NVRHI queue submission instance to its queue-keyed token; completion is observed through that queue's NVRHI timeline counter. Its bounded wait polls that counter only, so the adapter creates no second raw-Vulkan command or submission path. In both adapters, a command list records its own last token and `Begin` rejects allocator/recording reset until that token reports complete. `SubmitAndWait` is a compatibility wrapper over submit plus bounded completion wait so bootstrap consumers do not fork submission policy. CPU frame numbers and synchronous destroy-and-recreate are not acceptable substitutes because neither proves GPU retirement before reuse.

Physical graph binding also requires a backend-neutral resource/device ownership check before command recording. Resource descriptions prove shape and usage compatibility but cannot prove that a supplied texture or buffer belongs to the executor's device. `Engine::RHI::Device::OwnsResource(const Texture*)` and its buffer overload answer from an immutable, monotonic RHI-wrapper owner identity without leaking native handles or backend types. Null, foreign-backend, and same-backend cross-device resources fail binding before any pass callback or transition is recorded. The identity is allocated per creating RHI device and is not a native-pointer comparison, so a later device cannot become compatible through address reuse. A resource must remain live for the query; passing a destroyed wrapper is invalid caller behavior, and no ownership query is valid after its device has been destroyed. Description-only checks and native downcasts above RHI are rejected because they either admit invalid cross-device work or break backend neutrality.

Portable execution-core evidence requires the same backend-neutral offscreen readback boundary on every claimed backend. Vulkan/NVRHI already implements `RHI::Device::ReadbackTexture`, while D3D12 originally returned failure and the presentation bridge owned a separate native capture path. Phase 3 therefore requires D3D12 RHI readback parity before graph execution can be checked. The RHI path validates exact device ownership, supported format/extent/state, copy row pitch and footprint, submission, bounded GPU completion, and returned byte layout; it must not call or duplicate presentation/ImGui policy. Native viewport capture is regression evidence for presentation, not a substitute for graph-owned resource output.

Imported-resource initial-state validation requires the graph binder to compare its declaration with the owning RHI wrapper's authoritative tracked state before recording. Immutable resource descriptions cannot provide that evidence. `RHI::Device::QueryResourceState` therefore exposes a narrow backend-neutral observation for exact-device-owned live textures and buffers. The query returns only `ResourceState`, fails for null, foreign-backend, same-backend other-device, or `Unknown` state, and never mutates or adopts state. It reads the wrapper's committed state, not a creation description, a native downcast above RHI, or a caller assertion.

State timing is explicit: while a command list records, its transitions remain command-list-local pending states so later barriers in that same recording use the correct source state, while `QueryResourceState` continues to report the last state accepted by `Device::Submit`. A valid queue acceptance token commits the pending wrapper states; failed recording, close, or submission does not publish a new state. D3D12 commits the wrapper's native whole-resource and portable state together; Vulkan/NVRHI commits its portable wrapper state after NVRHI accepts the closed list. The contract does not claim rollback of commands already accepted by a native queue after a device-loss failure. Native/external state adoption is intentionally absent until a real bridge specifies ownership, synchronization, and verification; callers may not assert a convenient state to bypass validation.

D3D12 now implements that boundary for one explicit portable shape: a live resource owned by the exact RHI device, with `R8G8B8A8Unorm`, one 2D mip/layer, one sample, nonzero extent, `CopySource` usage, and already-finalized `CopySource` state. Null/foreign wrappers, depth, unsupported format, unsupported subresource/sample shape, invalid footprint arithmetic, and any other state fail before a command list is submitted. The adapter obtains the D3D12 copyable footprint internally, records `CopyTextureRegion` into an RHI-created CPU-readback buffer on an RHI command list, submits through `CompletionToken`, and waits with the finite 5000 ms deadline. It does not transition the source, so its tracked state remains `CopySource` for the caller. Native row alignment remains internal: each valid row is copied into a compact CPU result with `RowPitchBytes == width * 4`, and D3D12 padding is never returned as bytes. This does not widen the public contract to D3D12 resources, footprints, states, fences, or presentation capture.

## Barrier Authority

The graph is the authority for intended logical resource states, pass dependencies, and queue ownership. `Engine::RHI` translates compiled transitions into backend commands. NVRHI automatic state tracking may remain a bootstrap implementation aid only under this rule:

- a graph-owned command list uses one declared mode for its whole recording lifetime;
- in automatic mode, the backend may ask NVRHI to emit barriers, but it must validate that the observed transitions match the compiled graph;
- in explicit mode, the graph-derived RHI transition batches are recorded and NVRHI automatic barriers are disabled for graph-owned resources/commands;
- a pass must never mix automatic and explicit ownership in a way that can emit duplicate, reordered, or contradictory transitions;
- imported-resource initial/final states and cross-queue signal/wait edges remain explicit regardless of backend automation.

Execution covers both textures and buffers. Because construction already emits buffer-state transitions, the backend-neutral command-list contract must expose explicit buffer transitions on D3D12 and Vulkan/NVRHI before the execution core can truthfully consume every compiled resource class. A texture-only executor is not an acceptable silent fallback; unsupported state/usage combinations must fail before submission with actionable diagnostics.

`CommandList::TransitionBuffer` records a whole-buffer transition from a compiled `ResourceState`; it accepts neither an inferred pass use nor a native state value. GPU-only buffers may transition to `Common`, compatible copy states, read-only `ShaderResource`, or storage `UnorderedAccess`; CPU-visible upload/readback buffers, `Unknown`, and texture-only states fail. D3D12 owns the native whole-resource barrier and tracked state, mapping vertex/index/constant read-only use to `GENERIC_READ`; Vulkan owns NVRHI `setBufferState` plus `commitBarriers` and tracks no-op repeats. These are backend translation details, not graph policy. Existing upload/copy helpers retain their scoped bootstrap barriers; when a destination is already explicitly `CopyDest`, they emit no duplicate transition.

The transition from bootstrap automatic barriers to explicit graph barriers is measured per backend. It does not change renderer pass declarations or permit native API state policy to leak above `Engine::RHI`.

## Queue Topology And Cross-Queue Submission

Vulkan/NVRHI admits only selected concrete family/index handles through the engine-created `VkDevice`. Portable submission resolves to its effective NVRHI queue; distinct effective dependencies become NVRHI timeline waits and completion remains queue-keyed. Ordinary work may use a resource across same-effective or same-family queues. Different-family ordinary use rejects; paired ownership is the sole admitted cross-family resource path and records the matched native translation described below. The RenderGraph executor resolves each compiled pass, submits stable producer-token fan-in, and records paired release/acquire only across distinct effective queues. Same-effective fallback retains ordered submission without fabricated ownership. Accepted submissions publish their prefix state/owner; failures publish neither staged state nor ownership. The execution result retains every accepted token and contexts retire per effective queue plus compiled pass identity.

Cross-queue execution distinguishes three facts that must not be collapsed: a queue family or graphics queue may support a command capability; the device may enable an independent queue of that class; and the graph may resolve a requested class to the graphics queue as an explicit fallback. Capability flags, non-null aliased native queue pointers, and independent queue availability are not interchangeable. The RHI publishes deterministic effective-queue resolution for `Graphics`, `Compute`, and `Copy`, including whether the result is independent or a graphics fallback.

The RHI GPU-dependency boundary accepts only prior valid completion tokens issued by the same device. A consumer submission waits on those producer tokens on the GPU; it does not call the CPU completion wait as a substitute. Invalid, foreign-device, unissued, forward, cyclic, and impossible self dependencies fail before submission. When producer and consumer resolve to the same effective queue, existing queue order satisfies the dependency and no artificial signal/wait is emitted. D3D12 maps independent-queue dependencies to queue fence signal/wait ordering while retaining native objects behind RHI. Vulkan/NVRHI maps selected independent queues to NVRHI timeline waits and retains Graphics fallback when no concrete compatible queue handle was admitted.

The implemented token model assigns the prospective consumer identity monotonically. Dependencies are validated in caller order before any backend wait or command-list submission: zero and foreign identities fail first; the prospective identity is an impossible self edge; larger identities are forward edges; repeated prior identities are duplicates; and absent smaller identities are unissued. Requiring every accepted edge to target an already issued smaller identity makes cycles unrepresentable rather than detecting them after native submission. Multiple valid dependencies retain caller order. Validation failure leaves the command list closed with staged resource states unpublished and issues neither native work nor a completion token.

D3D12 creates command lists for the effective queue class while preserving the requested portable class. Independently created Copy/Compute queues are identified by distinct native queue identity, never by a merely non-null pointer that may alias Graphics. A distinct consumer queue calls its GPU queue `Wait` on the already-signaled producer fence/value before executing the consumer; no CPU completion wait occurs between submissions. A same-effective-queue edge emits no wait. The forced-fallback diagnostic mode suppresses independent Copy/Compute creation so the identical portable recording and dependency path is exercised through ordered Graphics submission. Accepted submissions retain independently queryable queue-local fence tokens and commit staged states only after signal/token acceptance.

Vulkan topology records selected queue and family identities. Distinct queues in one family require GPU ordering but no queue-family ownership transfer. Different families require semaphore/timeline ordering plus matched release/acquire ownership transitions. Selected different-family queues admit a shared resource only through the paired ownership path; arbitrary ordinary use still rejects. Hosts without a concrete compatible queue retain documented Graphics fallback and must not report cross-family qualification. NVRHI queue wait/semaphore state remains owned and serialized by the adapter; graph or renderer code never mutates it directly.

NVRHI's Vulkan state tracker emits ordinary image/buffer barriers with ignored queue-family indices, so it cannot by itself authorize shared-resource movement between different families. The adapter therefore records raw matched native barriers only inside its RHI escape hatch, then reseeds NVRHI tracking to the resulting portable state without publishing the pending wrapper state. The dependency-ordered implementation was: portable paired buffer lifecycle and deterministic state/publication tests; real D3D12 independent/fallback qualification; texture parity under the same authority; selected Vulkan queues for queue-local work; then native different-family translation and split-family qualification. This order avoids a fake-only interface, an unbounded multi-backend slice, and a circular requirement to exercise queues before they exist.

A source-queue command list records a release and a destination-queue list records the matching acquire for the same exact-device-owned live resource, resolved queues, and portable before/after states. The destination submission depends on the accepted source token. Null/foreign resources, same effective queue, mismatched pairs, wrong committed owner/state, missing dependency, duplicate release/acquire, and failed source acceptance are rejected before publishing a new owner or state. State/owner publication occurs only after the acquire submission is accepted; an accepted release with a failed acquire remains a real pending-transfer state that must be retired or recovered explicitly rather than rolled back in prose.

`BufferOwnershipTracker` and `TextureOwnershipTracker` are separate per-RHI-device production authorities with identical paired-lifecycle semantics. They register only live exact-device wrappers with their committed portable owner/state. Command-list release/acquire methods stage ordered private operation vectors; duplicate resources of the same kind in one list reject at record time and no staged operation mutates a tracker. Submission validates every staged operation before any native wait or acceptance, then publishes every accepted operation in recorded order. A valid accepted release attaches the exact newly issued completion token to one pending pair while making ordinary transition, upload/copy where applicable, destruction, state/owner query, and second release unavailable. A valid acquire must name the exact pending token and include it exactly once among the already validated submission dependencies. Only accepted acquire publication clears pending and commits destination owner/state. Failed recording or complete-batch dependency/ownership validation leaves the prior public state or pending transfer intact; an accepted native submission is never reported as an unaccepted rollback.

Abandoned resource recovery is an explicit forward operation, not rollback. The device accepts it only for the same live buffer or texture and exact release token after `QueryCompletion` reports that token complete. For Vulkan different-family native releases, recovery first consumes a backend-private immutable pending snapshot and performs a GPU-ordered destination acquire of the original pair, destination-to-source release, and source acquire; only after that bounded compensation succeeds may the tracker clear pending and restore the recorded source owner/before-state. Same-family recovery has no family barriers but retains the same exact-token rule. Zero, foreign, incomplete, failed, or independently retired tokens cannot recover another transfer. Resource destruction unregisters the live wrapper only when no transfer is pending; destroying a pending wrapper is a contract violation rather than implicit cancellation.

For Vulkan different-family transfers, the RHI implementation obtains native command buffer and resource handles only inside its adapter and emits matched release/acquire barriers with the selected source/destination family indices. This is an RHI backend translation escape hatch, not a raw-Vulkan renderer path. Same-family distinct queues use the GPU dependency only and emit no family ownership barrier. D3D12 has global resource ownership across queues, so the family-ownership portion collapses while portable state/dependency validation remains. Local RTX 3080 Ti split-family evidence qualifies buffer and texture shared-resource movement; hosted Vulkan remains fallback/regression evidence unless it provides an exercised independent different-family pair.

The current D3D12 production adapter therefore emits no barrier while recording release. Its matching acquire records the portable whole-resource state transition on the destination list; D3D12 queue submission enqueues the producer-fence wait before executing that list, so the transition executes after the GPU dependency. Texture transfers are explicitly limited to one nonzero-extent 2D mip, array layer, and sample; multi-mip/layer/sample shapes and incompatible usage/state are rejected rather than silently transferring a subresource. `RHIBufferOwnershipSmokeV1` qualifies the local RTX 3080 Ti D3D12 and Vulkan split-family paths: it accepts Graphics-to-independent-Copy release/acquire with no CPU wait between submissions, moves bytes through a GPU-only validation destination on Copy, returns that destination to Graphics, then copies to CPU readback, observes exact 4 KiB bytes and final Copy owner/CopySource state, retires all issued tokens, and recovers a separate completed abandoned release. `RHITextureOwnershipSmokeV1` qualifies the matching texture path with exact release-token dependency, deterministic RGBA8 output through owner-queue `ReadbackTexture`, final Copy owner/CopySource state, retirement, and completed abandoned-release recovery. Forced Graphics fallback still rejects release recording.

Cross-queue graph execution uses per-submission/per-effective-queue recording contexts and retires every accepted context by its own completion token. A frame spanning several submissions needs an aggregate completion result rather than pretending one queue token covers all work. Imported state publication is accepted-prefix authoritative: if an early queue submission is accepted and a later one fails, already accepted resource states and tokens remain real and cannot be rolled back. The executor stops later scheduling, reports the accepted submission/pass prefix and failure, and retires every accepted context. CPU serialization is permitted only as a separately identified unavailable-queue fallback; normal same-graphics fallback relies on ordered submission and needs no artificial wait.

## Transient Resource Contract

Transient allocation depends on the compiled graph above. Reuse is legal only when lifetimes do not overlap in resolved execution order and resource compatibility requirements are satisfied. Cross-frame reuse must be gated by GPU retirement, not CPU frame number. Aliasing or ownership barriers required by the backend must be emitted by the compiled graph.

Heap placement/aliasing is an optimization, not a correctness requirement. A backend/device that cannot safely use aliasing heaps must fall back to separately allocated or non-aliased pooled resources while preserving the same graph lifetimes and results. Diagnostics must report the selected allocation mode and its memory cost; the hosted MoltenVK `MTLHeap` workaround must not become an assumed production capability.

`RenderGraph::BuildTransientResourceCapabilityGroup` publishes `Phase3TransientResourcesV1`. Current D3D12 and Vulkan/NVRHI report placed resources and alias barriers unavailable, so execution selects `NonAliasedGpuRetiredPool`. Unbound transient declarations bind to compatible RHI objects; sequential non-overlapping lifetimes share an object only on one effective queue and only when the prior final state equals the next declared initial state. Other overlaps, queue combinations, or state hand-offs allocate separately. At every successful pass submission, its exact token is immediately attached to every touched allocation; a later failure retains those accepted-prefix tokens and cannot expose in-flight storage. `ExecuteResult` reports mode and estimated logical allocated/pooled bytes, never native heap footprints. A selected placed/aliased mode fails explicitly until RHI supplies both placed-resource creation and alias-barrier emission.

This is why the Phase 3 transient-allocation item follows frame/render graph construction.

## Downstream Dependencies

No roadmap items were reordered beyond inserting the missing prerequisite immediately before transient allocation. The following unchecked work implicitly consumes the graph and must not create independent pass schedulers:

- Phase 3: scene-viewport resource/command integration, mip generation, Forward+/clustered lighting, PBR, lights, shadows, sky/atmosphere, debug overlays, and GPU timing/capture scopes.
- Phase 4: the multi-pass coverage, antialiasing, filtering, and native-resolution validation pipeline.
- Phase 6: visibility, occluder, HZB, material-resolve, compact G-buffer, and debug-view passes.
- Phase 7: runtime GPU culling, LOD selection, transitions, and optional runtime geometry paths. Offline meshlet/cluster cooking does not depend on the frame graph.
- Phase 8: runtime probe, lightmap, sky, reflection, and indirect-lighting passes.
- Phase 9: the full sparse-ray classification, tracing, reconstruction, composition, and debug pipeline.
- Phase 15: GPU timing, memory/transient-pool inspection, render debug views, captures, and golden render tests.
- Phase 16: optional upscaler integrations when they are expressed as renderer passes.

These are dependency flags, not claims that every listed item requires a unique graph feature.

## Completion And Verification

The Phase 3 construction item is complete when focused deterministic tests cover declared-use validation, dependency ordering and independent-pass stability, cycles, read-before-write, lifetime calculation in resolved order, barriers, and cross-queue dependency records, with the normal build/test suite and code style passing.

The D3D12 readback prerequisite is complete only with a real D3D12 offscreen clear/copy/readback smoke. Its three-by-two RGBA8 target intentionally requires a 256-byte native copy footprint while the returned result has the compact 12-byte row pitch and exactly 24 initialized bytes. The smoke must reject a same-device texture not finalized to `CopySource` and an otherwise valid CopySource R8 texture, then require every returned pixel to equal the deterministic clear. `TestRender.ps1` consumes the `RHITextureReadbackSmokeV1` marker; `TestVulkan.ps1` remains the Vulkan/NVRHI readback regression evidence. Neither smoke is graph execution, cross-queue scheduling, transient allocation, or viewport adoption evidence.

The following execution-core item separately requires imported and explicitly supplied physical resources to bind to real RHI resources, callbacks and graph-derived same-queue RHI barriers/commands to execute, deterministic submission, GPU retirement, and focused real-resource evidence. The next cross-queue item requires compiled queue dependencies to become explicit RHI signal/wait submission and ownership transitions; a queue-transition record is intentionally not an emitted synchronization primitive. The completed real-workflow path builds the representative multi-pass Scene viewport on D3D12 and Vulkan with graph/pass labels. Under the graph smoke flag only, it creates separate renderer-owned color/depth outputs and replays the former direct clear/raster recorder with the same immutable prepared frame, clear, pipeline, resources, constants, and draw order. The graph output is temporarily copied/read back and restored to `ShaderResource`; the reference ends in `CopySource` for the same backend-neutral readback. Equal extent, row pitch, and every returned byte are required. The reference output is never published to the viewport or native presentation bridge, and a mismatch fails with `SceneViewportRenderGraphV1` including backend, exact-byte result, dimensions, and byte count. The oracle remains retained after proof; transient allocation and reuse remain separate.

The completed execution core keeps that boundary narrow: every logical resource is explicitly bound to an exact-owner RHI wrapper with an identical kind, shape/size, usage, and initial-state description, then its authoritative committed state is matched before recording. It accepts graphics-only compiled passes, emits each pass's compiler-owned same-queue barriers before its callback, and gives the callback only its declared texture/buffer handles plus the assigned RHI command list. Missing/invalid bindings and imported-state mismatches fail before recording; undeclared or wrong-kind context lookup returns null. Callback failure or exception discards that recording and invokes no later callback; transition, close, submission, and completion-query failures likewise prevent success.

The executor owns a fixed three-context pool. Each successful queue acceptance stores the returned opaque token on that exact context. Acquisition selects only an unused context or one whose token query reports complete; three incomplete tokens produce explicit exhaustion, while invalid/failed completion status stops execution. Failed, unsubmitted recordings are destroyed rather than reused. Successful submission is the sole state-publication boundary, after which imported resources must query at their compiler-derived final state. The execution result exposes the stable pool index and whether a retired context was reused for diagnostics and evidence, not as scheduling policy. Cross-queue records, transient allocation/reuse, and viewport migration remain excluded.

The queue-topology/GPU-dependency prerequisite is verified independently from graph execution. Deterministic tests cover requested/effective/independent resolution, one and multiple prior dependencies, same-effective elision, distinct-effective GPU-wait selection, every invalid token class, no submission or state publication on validation failure, and independent token retirement/queryability. `RHIQueueDependencySmokeV1` on D3D12 records a 4 KiB upload-to-intermediate copy on Copy, submits it, records the intermediate-to-readback consumer on Graphics, and submits with the producer token without an intervening CPU wait. It validates exact bytes and committed `CopySource` state, then chains Graphics to Compute when independently enabled. The forced-fallback run proves both edges use ordered Graphics submission. Vulkan retains its one enabled Graphics queue and proves truthful Copy/Compute fallback plus same-effective ordered token dependencies without adding buffer-copy or multi-queue claims.

`Phase3TransientResourcesV1` plus allocation is verified by deterministic lifetime/compatibility/peak-cost and exact-token tests, and by `RenderGraphTransientAllocationSmokeV1` on the current D3D12/Vulkan fallback. This is not native placed-resource, alias-barrier, hosted, or macOS evidence.
