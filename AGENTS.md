# Workspace Agent Instructions

These instructions are the persistent operating contract for any AI or human agent working in this repository. Do not rely on chat memory, screenshots, or another agent's summary when repository evidence is available.

## Start Here

Before roadmap, architecture, or implementation work:

1. Read this file.
2. Read [PLAN.md](PLAN.md) for current state and execution order.
3. Read [Docs/README.md](Docs/README.md) for the complete documentation catalog, authority rules, and update responsibilities.
4. Read [Docs/ROADMAP_GOVERNANCE.md](Docs/ROADMAP_GOVERNANCE.md) before changing roadmap wording or checkboxes.
5. Read [Docs/TESTING_STRATEGY.md](Docs/TESTING_STRATEGY.md) before creating, changing, or delegating nontrivial tests.
6. Read [Docs/VERIFICATION.md](Docs/VERIFICATION.md) before claiming behavior is complete.
7. Read the task-relevant architecture contracts linked by [Docs/Architecture/README.md](Docs/Architecture/README.md) and the nearest `OWNERSHIP.md` before editing a subsystem.

For a broad architecture audit, read every document listed in `Docs/README.md`. For a bounded implementation, read the mandatory set above plus every contract named for that subsystem; do not substitute a stale “first implementation order” in a research document for `PLAN.md`.

## Documentation Authority And Conflicts

- `PLAN.md` alone owns current implementation state, roadmap order, and completion checkboxes.
- `PRODUCT.md` owns product goals and user-facing principles. `DESIGN.md` owns editor visual and interaction rules.
- `Docs/Architecture/*` owns technical decisions and implementation contracts. ADRs record choices and their consequences; research/reference documents do not override accepted contracts.
- `Docs/DEPENDENCIES.md` owns admitted dependency versions, licenses, and integration boundaries.
- `OWNERSHIP.md` files own directory/module scope and forbidden dependencies.
- Source code, tests, runtime captures, and CI are the evidence of actual behavior. When prose disagrees with evidence, do not rationalize the mismatch: report it and update the relevant current-state or contract document in the same scoped change.
- A user instruction can change project direction. If that change is intended to persist, update the repository documents that future agents will read; do not leave the decision only in chat.
- If two authoritative documents conflict and the intended resolution cannot be established from code, tests, roadmap order, or an accepted ADR, stop and flag the decision instead of silently choosing one.

## Research Fidelity And Mechanism Preservation

- Treat user-supplied chats, transcripts, screenshots, diagrams, and external critiques as source material. Before compressing them into a decision, extract the named mechanisms, operation order, timing/control point, defaults, alternatives, trade-offs, caveats, and unresolved claims.
- Do not replace a specific mechanism with a broader label when that loses the reason it was supplied. For example, an RTSS-ASYNC-inspired inter-frame wait, a submission gate, Front Edge Sync immediately before `Present`, and Reflex-style latency control are distinct hypotheses; “frame pacing” is not an adequate substitute for documenting their different control points.
- Separate four evidence levels in durable docs: the source's claim, the agent's inference, the accepted project decision, and behavior verified in this repository. Community posts and assistant chats may motivate a bake-off, but they are not implementation facts until confirmed by primary documentation, code inspection, or engine-owned measurement.
- When a supplied source changes project direction, write the mechanism and its provenance into the authoritative architecture contract and put implementation/verification work in `PLAN.md`. Do not leave the only recoverable version in chat or reduce it to a generic summary future agents can misread.
- Before handoff, compare the edited contract element-by-element with the supplied source. If a named mode, ordering constraint, default, or warning was intentionally rejected, record that rejection and why instead of silently omitting it.

## Intent-First Review, Contract Challenges, And Minimality

- Begin review with the architecture, mechanism, invariants, failure behavior, and user-visible evidence. Prefer a concise implementation explanation plus targeted questions over an indiscriminate code dump. This reallocates attention; it does not prohibit source inspection.
- Inspect source directly when risk warrants it or evidence and explanation disagree. Synchronization/lifetimes, unsafe/native boundaries, serialization/migration, security/permissions, persistent data loss, and performance-critical allocation or scheduling require proportionate implementation review.
- Accepted contracts remain authoritative but are challengeable by evidence. Pause the affected slice, record the contradiction and reproducible evidence, identify the affected invariant, propose the smallest coherent contract change with alternatives/migration/verification impact, obtain the required authority, update the authoritative document, then resume. Never silently route around a contract.
- Prefer the smallest mechanism satisfying a concrete current requirement. New abstractions, modules, extension points, or dependencies must name a current consumer, the simpler alternative considered, the behavioral/verification boundary they create, and how they can be removed or replaced. Do not add generic infrastructure for hypothetical consumers or duplicate an existing authority.
- Expose concise plans, progress, mechanism decisions, tool actions, and evidence so the user can interrupt, correct, and resume work. Do not equate explainability with disclosure of private chain-of-thought.
- Architecture-changing commit messages must explain what changed and why, motivating evidence/source, important rejected alternatives, compatibility/deferred work, and verification.
- A durable handoff must leave the objective, accepted decisions, actual current behavior, evidence, limitations/blockers, working-tree state, and next ordered work recoverable from repository files. After compaction or a new chat, reread the mandatory documents and task-relevant contract rather than trusting a summary.
- The complete repository-agent and future product-agent decision lives in [Docs/Architecture/AI_AUTOMATION_ARCHITECTURE.md](Docs/Architecture/AI_AUTOMATION_ARCHITECTURE.md).

## Verification

- Follow [Docs/TESTING_STRATEGY.md](Docs/TESTING_STRATEGY.md) for test design and [Docs/VERIFICATION.md](Docs/VERIFICATION.md) for commands/evidence. Use the smallest test tier that exercises the changed behavior plus proportionate regression coverage.
- Tests normally assert public or deliberately test-facing behavior, while implementation knowledge selects fragile boundaries, states, and failure sequences. Generated tests require an invariant or independent oracle, reproducible seed/trace, failure artifact, minimization path, and explicit runtime tier; line coverage is diagnostic, not completion evidence.
- Never delegate tests with only “write tests.” Give the agent the behavior/public contract, implementation/control boundaries, fragile limits, invariant/oracle, generator or corpus distribution, corruption strategy, replay/minimization requirements, tier/budget/sanitizer mode, required commands, and exact platform/backend non-claims.
- For frame pacing or latency work, never accept one hook-local frametime graph as evidence. Carry one frame ID through engine frame start, input/simulation, render submission, `Present`, GPU completion, and display feedback; report start-to-start cadence, active work, intentional pacing wait, present cadence, and display cadence separately. A limiter/overlay graph sampled near its own delay point may look flat while visible delivery still stutters.
- When a completed feature can be exercised locally, test the new behavior itself rather than relying only on compilation.
- Verification scripts that launch headed child processes must bound each child runtime independently of build time, stream live stdout/stderr while preserving assertion output, and report timeout diagnostics plus verified process-tree cleanup. A cancelled or timed-out hosted run is failed evidence, never a pass.
- For editor-facing changes, run the editor and inspect a screenshot when practical. Use an existing automated smoke test when it covers the interaction; otherwise add focused coverage that does.
- For platform/backend claims, run that backend. If local hardware is unavailable and the repository has matching hosted CI, push the scoped change and use the completed job as evidence.
- Agent reports, source inspection, successful compilation, and a matching log marker are not substitutes for the required runtime behavior.
- Report verification that could not be performed and the reason.

## Roadmap Integrity

- In `PLAN.md`, `[x]` means the exact behavior written on that line is implemented, integrated into its real workflow, and verified. Compilation alone is not enough for a runtime or editor-facing feature.
- Never check an item whose delivered artifact is only a stub, placeholder, skeleton, plan, scaffold, interface, or unused implementation. Put foundations in current-state prose and keep the behavioral item unchecked.
- Before changing a roadmap checkbox, update current-state prose, add or identify focused verification, run `Scripts/CheckCodeStyle`, and confirm the checked wording does not overstate platform, backend, device, or workflow coverage.
- Phase completion means every required item and exit criterion is demonstrably met. A phase may have useful checked foundations without being complete.
- Do not reorder unrelated roadmap items silently. Record new prerequisites immediately before their consumers and explain cross-phase dependencies in the relevant architecture contract.

## Agentic Workflow

- For requests to continue the roadmap, inspect the first unchecked `PLAN.md` item in order and implement that item rather than substituting an invented slice. If it must be split for honest platform or verification scope, preserve the original intent with precise completed and remaining wording.
- Before selecting a platform-specific first unchecked item, confirm that an execution environment capable of the required focused verification is available. Do not implement speculative platform behavior or treat cross-generation, source inspection, or a zero-step CI job as platform evidence; defer or reorder the item only with explicit user direction and a matching `PLAN.md` plus task-relevant contract update.
- Evaluate architecture and roadmap integrity before implementation. If the next item depends on missing infrastructure, document and schedule the prerequisite first; do not build dependent behavior on an implicit subsystem.
- Use the `terra_medium` custom agent for substantive delegated repository work. Its native subagent definition is `$CODEX_HOME/agents/terra_medium.toml` (normally `~/.codex/agents/terra_medium.toml`; alternate local accounts may use another `CODEX_HOME` such as `~/.codex_1`), whose `name = "terra_medium"` pins `model = "gpt-5.6-terra"`, medium reasoning, and the implementation-owner contract. `$CODEX_HOME/terra_medium.config.toml` mirrors those behavioral settings for the direct persistent-owner fallback below. Never use generic, default, worker, or inherited subagents when `terra_medium` is available; give it fresh scoped context instead of the complete parent history. Use as many Terra workers concurrently as there are concrete, independent, non-overlapping tasks that justify them; minimize token waste, redundant research or review, and idle agents, and never use concurrency as a quota. If neither the native custom-agent route nor the direct persistent-owner fallback is available, continue in the root session and report that routing is unavailable.
- Before starting a roadmap slice, classify it as small, medium, or large from its required subsystems, artifacts, and verification. If one checklist line proves to contain multiple large independently reviewable mechanisms, split explicit prerequisites into `PLAN.md` before continuing rather than consuming an unbounded session to satisfy the wording atomically.
- For medium/large work at synchronization, serialization, migration, native-boundary, or persistence risk, the owner must finish one read-only feasibility gate before editing: identify the exact control/publication boundaries, resolve any token/state/order dependency that could invalidate the proposed split, and name the canonical local build plus focused test commands from `Docs/VERIFICATION.md`/`Scripts`. Do not probe an unrelated build system such as CMake when the repository documents Premake/MSBuild or GMake entry points. If feasibility exposes a contradiction, report it before generating and reverting a substantial patch.
- When real backend/runtime evidence exposes a previously implicit architecture prerequisite, stop the parent mechanism, add the prerequisite immediately before it in `PLAN.md`, and finish that prerequisite as its own coherent verified slice before resuming the parent. Do not carry one growing dirty parent patch across replacement owners while simultaneously inventing its missing RHI, serialization, synchronization, or migration foundation; that destroys per-slice evidence and makes allowance cost opaque.
- Treat model allowance as a finite engineering resource. When reliable allowance telemetry is available, report the starting state for medium/large work and re-check only at meaningful milestones. At 10% remaining, do not begin a large slice or add optional reviewers; at 3% remaining, stop implementation, terminate optional agents, write or refresh `HANDOFF.md`, preserve the working tree, and return control unless the user explicitly authorizes spending the remainder on a named bounded action.
- The orchestrator owns human intent, product and architecture coherence, scope selection, assignment boundaries, live steering, conflict resolution, final acceptance evidence, and durable handoff. It controls the ideas and evidence, not every generated line.
- The default delegation pattern is one persistent exact `terra_medium` owner per coherent vertical slice. That owner implements the slice, runs focused verification, investigates failures, and reports the concrete diff plus evidence. If its result is partial or incomplete, resume that same owner with `followup_task` and one consolidated correction; replace it only when it is unavailable/blocked, the scope materially changes, or an independent specialist is justified, and record why in `HANDOFF.md`. Do not independently redo the same research, implementation analysis, or code audit assigned to that agent merely to feel informed.
- Add another agent only when it provides independent value: disjoint parallel work, a specialist risk audit, contradictory evidence, incomplete verification, security/concurrency/serialization/migration/platform/performance risk, or a user-requested independent review. Temporary specialist agents terminate when their finding or correction is complete.
- Prefer reviewing concrete work after it exists. Do not spend tokens having multiple agents audit hypothetical choices in parallel unless the architecture decision must be resolved before implementation.
- Give each agent only the smallest sufficient context: the relevant `PLAN.md` item, task-relevant architecture contract, nearest `OWNERSHIP.md`, relevant source files, required verification, known non-goals, and forbidden paths. Do not make every agent reread the whole repository by default.
- Require agents to report the useful review surface: selected mechanism and data/control flow, preserved invariants, rejected alternatives, real tests/captures/benchmarks/scenarios run, remaining uncertainty or blocked evidence, and files changed.
- Source inspection by the orchestrator is targeted: inspect when outputs conflict, behavior and explanation disagree, verification fails or is incomplete, high-risk boundaries are touched, or focused questions are needed for acceptance. Exhaustive second-pass code reading is not the default acceptance gate.
- Parallel agents must not silently edit the same files or own the same decision. Their scopes must be explicitly disjoint, or one must be read-only. Agents sharing this workspace must not run concurrent generators, compilers, formatters, or tests that write the same build/output tree.
- A user correction immediately supersedes the affected delegated plan. Stop or redirect only the affected work; do not restart unrelated completed work.
- The normal orchestrator loop is: assign scoped work, receive a mechanism/evidence summary, ask focused follow-up only if needed, reconcile conflicts, run shared verification, update durable repository state, and commit with rationale.
- Keep that loop coarse-grained. Per owner, normally send one complete assignment, at most one consolidated correction when evidence requires it, and one final acceptance exchange. Target at most 20 root model resumptions per coherent slice through pre-CI acceptance and at most two explicit waits per owner/process. A direct owner that reaches 20 sampled model steps, 1.5 million reported total tokens, or 120 seconds without a final result is workflow-failure evidence, not permission to keep polling or resuming it. Preserve its patch, stop it at the next safe boundary, inspect once, and continue the same authorized roadmap work with a narrower fresh owner or locally; do not ask the user to say "go on." Sessions equal coherent slices, and a completed-slice owner is never resumed for the next checkbox. Do not resume even for a correction when its latest sampled input context is at or above 100k tokens; use a fresh narrowly scoped correction owner instead. Batch related reads, edits, and diagnostics; prefer milestone-sized patches and focused tests over file-by-file prompting and test-after-every-edit churn.
- Before issuing tools, define the next coherent discovery or implementation milestone and gather the independent reads, searches, status checks, and diagnostics needed for it in one invocation or parallel batch when safe. Do not alternate a reasoning turn with one shell command when several known, independent commands can run together. Keep batches bounded and filter or summarize output at the source. Raw owner transcripts, full build logs, full test logs, and unbounded document dumps must go to ignored files under `output/`; the root transcript receives only the compact final owner report, startup-banner fields, failing assertions, and short final summaries needed for a decision.
- Treat a make-then-revert loop as failed workflow evidence even when the final tree is clean. Record the reason, edit/tool churn, canonical command miss, and retained useful finding; resume the same owner with one corrected mechanism rather than hiding the cost or starting a cold replacement. Usage audits must distinguish weekly-meter movement and goal telemetry from raw/cached model tokens.
- Prefer one reviewable patch per coherent mechanism over a sequence of file-by-file patches. Run focused tests when they prove the changed behavior, but defer shared builds, generators, formatters, and broad regression suites to coherent integration milestones instead of rerunning them after each small edit.
- Treat high cached-input share as evidence that prompt caching is working, not that replay is free. Optimize the number and shape of root turns: keep exploration and noisy diagnostics in scoped subagents, return compact evidence summaries, and avoid repeating information already recorded in repository documents or an owner report. A high cached share paired with material weekly-meter movement is a failed cost result even if the cache itself worked. Cached/raw token totals are diagnostics, not exact weekly-allowance accounting.
- Prefer first-class subagent orchestration when it exposes custom-agent selection, and select the exact role name `terra_medium`; selecting only the `gpt-5.6-terra` model is not equivalent because it omits the custom agent's developer instructions. When the app root cannot select a custom role, start the owner **directly** as one persistent CLI session with `codex exec -p terra_medium -s <least-privilege-mode> -C <workspace> -`; the `$CODEX_HOME/terra_medium.config.toml` profile loads the same model, medium reasoning, and developer instructions as the native role. Resume that direct owner with the profile/sandbox/workspace options **before** the subcommand: `codex exec -p terra_medium -s <least-privilege-mode> -C <workspace> resume <session-id> -`. A bare `codex exec resume <session-id> -` reloads the default root configuration and may silently switch the session to Sol. Omit `--ephemeral` and always use `-o <ignored-result-path>`; redirect the CLI's complete stdout/stderr transcript to a separate ignored log instead of returning it through the root tool call (PowerShell example: `... -o <result> - *> <transcript>`). After completion, read only the banner fields needed to verify `gpt-5.6-terra`, medium reasoning, workspace/permissions/session ID, plus the compact result file. If the banner is wrong, stop before accepting edits. Do not ask the direct owner to reread whole `AGENTS.md`, `PLAN.md`, or `Docs/VERIFICATION.md` files already represented in its task packet: give exact line-bounded contracts and require bounded `rg -n -C` or `Select-Object -Skip/-First` reads unless the task is explicitly a broad audit. Do not insert a Luna/default wrapper or ask the direct owner to spawn another agent. Record that rollout when auditing usage. Never use a user-owned app task/thread as a substitute unless the user explicitly asks for a separate or background task.
- At root context >=100k tokens or after two to three slices, finish/commit the coherent slice, refresh `HANDOFF.md`, and continue in this same task through Desktop compaction. After compaction reread mandatory state and continue without asking the user. Never create/fork/steer another task for rollover. Stop only for genuine CI/user input, the user's allowance floor, or an interface impossibility; never checkpoint mid-edit/test/owner or use compaction to broaden scope.
- After launching a long-running owner or process, send at most one user-visible start update and then remain silent until it exits, fails, or produces a decision-relevant event. Use the longest available event/process wait rather than narrating unchanged status. For a direct CLI fallback with no event-driven completion callback, make at most two 60-second waits. If it still has no final result, terminate the direct owner, preserve and inspect its partial patch once, record the cap event, and continue automatically with a narrower fresh owner or root completion; do not poll indefinitely or require the user to say "go on." A safely detached external event-driven monitor may continue without root supervision. Never terminate during a known external generator/compiler write; direct owners subject to this cap therefore do not run the shared build.
- At integration milestones, batch push/CI, query once by full SHA, and retain compact status only. If local evidence is sufficient, use event-driven monitoring, continue useful work, or record CI pending; wait synchronously only when hosted/platform evidence is the blocking acceptance gate. This never weakens required backend evidence.
- The orchestrator reviews the completed patch and required evidence before starting the shared build. It sends any one consolidated correction first, then runs one consolidated shared verification pass over the final source. Do not build merely to begin review and then rebuild after an avoidable source/test correction. When the direct Terra sandbox has already proven the duplicate `Path`/`PATH` MSBuild failure in this workspace, do not make every new owner retry it; the owner runs bounded source/style/diff checks and root owns the one canonical build/test pass. Repeat research, audit, build, or test only for a concrete contradiction, missing required evidence, high-risk boundary, or integration failure; reassurance is not a reason to duplicate work.
- Amortizing a build never amortizes evidence. Every slice needs its own focused behavior test or capture and independently supportable roadmap wording. If the consolidated build or a focused test fails, isolate the responsible slice and fix or remove it from the batch without checking it off; do not weaken tests or hold unrelated verified slices hostage.
- Prefer separate commits per independently reviewable slice after the integrated batch passes, followed by a documentation/checkmark commit when hosted evidence is required. A single commit is acceptable when the slices form one atomic contract, but the handoff must still report each slice's evidence separately.
- Roadmap-continuation requests authorize staging the scoped changes, committing on the current branch, pushing to the configured remote, monitoring resulting CI, and fixing/retrying in-scope failures. This does not authorize unrelated changes, destructive history edits, releases, or deployment outside existing CI.
- Monitoring authorization is not a requirement to synchronously consume model allowance until CI completes. A pushed run may be handed off as pending with its exact URL/ID and qualification limits when cheap event-driven monitoring is unavailable.
- For a medium/large handoff with available telemetry, record per-slice root resumptions, explicit waits, Terra sessions versus coherent slices, cache share and context median/max, allowance start/end, and every exception reason in `HANDOFF.md`. Before handoff, ensure the working tree contains no accidental generated files and that every durable decision from the task is present in the appropriate Markdown file.
